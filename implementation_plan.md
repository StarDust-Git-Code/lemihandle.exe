# Spatial Intent Engine: Phase 3 Integration Plan

This document outlines the final integration step required to connect the frontend overlay (Developer A) with the multimodal AI backend (Developer B).

## The Overall Vision: What are we building?

We are building an **ambient, multimodal OS assistant**. 
Instead of typing queries into a chatbot window, the user interacts with the AI "spatially" while working. 

**How it works:**
1. **The Intent:** The user is looking at something on their screen (code, an email, an image) and needs help or action.
2. **The Trigger:** Instead of leaving their workflow, the user opens their palm to speak (or opens their mouth), asks a question ("Can you explain this function?"), and closes their palm (or closes their mouth / nods their head).
3. **The Payload:** The Frontend (Dev A) instantly captures a base64 screenshot of the user's active screen and transcribes their audio query.
4. **The Brain:** This payload (Screen + Voice) is sent to the Backend (Dev B). The Gemini AI analyzes the *visual context* of the screen to understand what the user is referring to in their voice query.
5. **The Action:** The backend executes any necessary local system tools (like saving a file) and sends a response back.
6. **The Result:** The Frontend displays the AI's response in a non-intrusive, glassmorphic overlay card that auto-dismisses when the user looks away.

The end result is an AI that "sees" what you see, "hears" what you say, and responds directly over your workflow without requiring you to switch applications.

## Current State

### Developer A (Frontend) - **COMPLETE**
The entire Spatial Intent Engine UI and sensor array is finished and production-ready:
- **UI:** Click-through transparent window, state transitions, animated glassmorphic result cards.
- **Vision:** Hand tracking (Pinch, Palm, Fist) and Face tracking (Head Nod, Head Shake, Jaw PTT, Frustration, Gaze tracking).
- **Audio:** Thread-safe background recording and Google Speech-to-Text transcription.
- **System:** System tray hardware toggles (Mic/Camera), `mss` screen capture, `Ctrl+Shift+Q` panic quit.
- **Network:** Non-blocking async POST layer.

**Current Blocker:** The frontend is running in `MOCK_MODE = True`. It captures the screen and voice perfectly, but instead of sending it to the backend, it simulates a response by reading a local `mock_response.json` file.

### Developer B (Backend) - **COMPLETE**
The FastAPI orchestration layer is finished:
- **API:** Accepts the `audio_transcript` and `screen_b64` JSON payload.
- **Model:** Uses `gemini-2.5-flash-lite` for ultra-low latency multimodal processing.
- **Memory:** Maintains contextual `CHAT_HISTORY` across turns.
- **Tools:** Supports local function execution (e.g., `save_output_to_file`).
